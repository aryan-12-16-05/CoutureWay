# Migration Notes — Supabase → Flask REST + SQLAlchemy

## What changed

**Removed** (Supabase backend, fully replaced):
- `src/integrations/supabase/*` (client, server client, types, auth attacher/middleware)
- `src/integrations/lovable/*` (Lovable Cloud OAuth helper)
- `src/lib/rest-helpers.ts`, `src/lib/config.server.ts`, `src/lib/api/example.functions.ts`
- `src/routes/api/*` (products/orders/appointments TanStack server routes)
- `supabase/` directory (config + SQL migrations — schema now lives in SQLAlchemy models)
- `@supabase/supabase-js` and `@lovable.dev/cloud-auth-js` from `package.json`

**Added:**
- `flask-backend/` — complete clean-architecture Flask REST API (see its README)
- `src/lib/api-client.ts` — JWT-aware fetch client; the only place the frontend talks to the network

**Modified (data layer only — zero JSX/styling changes):**
- `src/lib/auth-context.tsx` — same exported interface, now backed by the Flask API
- `src/routes/auth.tsx` — submit handlers call `auth.signIn/signUp`; Google button shows an explanatory toast
- `src/routes/_authenticated/route.tsx` — guard validates the JWT via `GET /auth/me`
- `src/routes/_authenticated/dashboard.tsx` — react-query fetchers hit Flask; realtime → 30s polling
- `src/routes/_authenticated/booking.tsx` — `POST /api/appointments`
- `src/components/notifications-bell.tsx` — unread-count endpoint + 30s polling
- `src/start.ts` — Supabase auth middleware removed

Everything else (all pages, components, styles, configs) is byte-identical to the original repo, so the UI renders exactly as before.

**Optional — unused shadcn/ui library files:**
The original repo contains ~45 stock shadcn/ui components in `src/components/ui/` that no page
imports (only `sonner.tsx` is used, and it is included here). The app builds and renders

`src/components/ui/` folder from Lovable, or re-add per component with
`npx shadcn@latest add <component>` (the `components.json` config is included).

**Not carried over (Lovable-internal):** `.lovable/`, `.env` (recreate from `.env.example`).

## Functional parity checklist

- ✔ Sign up / sign in (email+password, JWT) — profile + role auto-created, `ADMIN_EMAIL` gets admin
- ✔ Auth-guarded routes (/dashboard, /booking)
- ✔ Booking wizard creates appointments → confirmation notification appears in dashboard + bell
- ✔ Dashboard: orders, appointments, measurements (upsert), notifications, profile update
- ✔ Products API with category/limit filters + admin CRUD; seed data identical to old DB seed
- ✔ Orders API with line items, totals computed server-side, admin status updates → notifications
- ✔ Cart endpoints (schema parity; UI does not use them yet, same as before)
- ✔ Global JSON error handling: 400/401/403/404/405/409/422/500 with rollback
- ✖ Realtime push → replaced with 30s polling (Flask has no websocket layer by default)
- ✖ Google OAuth → requires configuring an OAuth provider against the Flask backend; button explains this
